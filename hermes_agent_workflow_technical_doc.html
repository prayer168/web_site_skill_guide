Hermes Agent 工作流專家級技術文件

從安裝、模型設定、工具權限、記憶與技能，到 Cron 自動化、Telegram／Open WebUI 整合與教育現場應用範例，建立一套能「越用越懂你」的個人 AI Agent 工作流。

本文中的 Hermes Agent 指 Nous Research 開源專案 Hermes Agent，不是 Hermès 品牌，也不是其他同名學術工具。

核心結論

Hermes Agent 最值得發揮的地方，不是一次性聊天，而是把重複任務沉澱成 Memory（記得什麼）、Skills（知道怎麼做）、Cron（定時去做）、Gateway（在哪裡回報） 的長期工作流。

1. Hermes Agent 的定位：不是聊天機器人，而是可累積能力的工作流 Agent

自我改進

官方將 Hermes Agent 定位為具備 closed learning loop 的自我改進 Agent：可從任務經驗建立技能、改善技能、提示自己保存知識，並跨 session 搜尋過去對話。

工具型 Agent

它不是只生成文字，而是可使用 web search、browser、terminal、file editing、memory、delegation、scheduled tasks 等工具組。

長駐服務

Hermes 可以在本機、WSL2、VPS、Docker、SSH、Modal/Daytona 等環境運行，再透過 CLI、桌面 App 或訊息平台互動。

適合你的場景：如果你常做教材網站、科展前置研究、教案文件、GitHub Pages 部署、定期蒐集教育／AI／自然科資料，Hermes Agent 的價值在於把「每次都要重新說明的流程」變成可重用的技能與排程。

2. 安裝：桌面版、CLI、WSL2 與伺服器部署

2.1 建議安裝路線

使用者類型建議路線理由
一般使用者Windows/macOS 桌面安裝器同時安裝桌面與 CLI，降低環境設定門檻。
Windows 技術使用者WSL2 + Linux 安裝指令適合搭配 Git、Node.js、Python、Codex/Claude Code 類工作流。
長駐 AgentVPS / Docker / SSH 後端可讓 Agent 不依賴個人電腦開機，適合 Telegram bot、排程摘要、伺服器監控。
研究與本地模型Linux + Ollama/vLLM/OpenRouter/Nous Portal方便測試多模型、切換 provider、控制成本與資料邊界。

2.2 官方快速安裝指令

官方文件指出，macOS 或 Windows 建議使用 Hermes Desktop installer；若只要命令列安裝，可使用下列指令。

Linux / macOS / WSL2 / Android Termux

curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash

Windows PowerShell 原生安裝

iex (irm https://hermes-agent.nousresearch.com/install.ps1)

安裝桌面版

hermes desktop
安裝器會做什麼？官方安裝文件說明，installer 會自動處理 Python、Node.js、ripgrep、ffmpeg、repo clone、virtual environment、全域 hermes 指令與 LLM provider 設定。

2.3 安裝後檢查

hermes --version
hermes doctor
hermes
  • 看到版本號：代表 CLI 已進入 PATH。
  • hermes doctor 無重大錯誤：代表依賴與環境大致正常。
  • 能啟動聊天：再進行模型、工具、Gateway 與 Cron 設定。

3. 首次使用：先讓一個乾淨對話跑起來

重要原則：不要一開始就同時設定 Telegram、Open WebUI、MCP、Cron、Docker。官方 Quickstart 的建議是：如果 Hermes 連普通對話都無法完成，不要先加更多功能；先讓基本聊天穩定。

3.1 選擇模型 Provider

hermes model

最省事的路線是使用 Nous Portal，官方文件說 hermes setup --portal 可用一次 OAuth 設定模型 provider 與 Tool Gateway，包括 web search、image generation、TTS、cloud browser。

hermes setup --portal

3.2 試跑基本任務

hermes chat -q "請用三點整理 Hermes Agent 的用途"
hermes chat -q "幫我檢查目前資料夾的檔案結構，並提出整理建議"

3.3 常用 CLI 模式

指令用途適用情境
hermes進入互動模式日常聊天、長任務討論
hermes chat -q "..."一次性任務腳本、CI、快速摘要
hermes chat --toolsets web,terminal,skills指定工具集控制任務可用能力

hermes chat --ignore-rules略過 AGENTS.md、SOUL.md、記憶與預載技能隔離測試、重現 bug

4. Hermes Agent 工作流總架構

1. 指令CLI／Desktop／Telegram／WebUI
2. 脈絡SOUL.md、AGENTS.md、Memory、@ references
3. 工具Web、Terminal、Files、MCP、Browser
4. 技能可重用流程、專案 SOP
5. 回饋結果交付、記憶更新、Cron 下次執行

4.1 推薦的「五層工作流」

層級目的實作方式範例
Identity定義 Agent 的語氣、角色、預設規範~/.hermes/SOUL.md「你是自然科教師的教材開發助理，使用繁體中文。」
Project Context讓 Agent 理解某個專案AGENTS.md、.hermes.md、CLAUDE.md教材網站架構、部署規則、檔名慣例
Memory保存穩定事實MEMORY.md、USER.md使用者偏好 RWD、繁中、GitHub Pages
Skills保存可重複流程~/.hermes/skills/ 或 skill_manage「產出科展前置作業」「生成教案與學習單」
Automation讓工作定期發生cronjob、hermes cron、Gateway每日 AI 教育新聞摘要、每週網站 broken links 檢查

5. Memory 與 Skills：Hermes 工作流真正變強的關鍵

Memory：記得「什麼」

適合放穩定事實、偏好、專案位置、常用工具與環境。官方最佳實務明確建議：Memory 是 facts，例如環境、偏好、專案位置。

請記住：我的教材網站通常使用繁體中文、RWD、單頁 HTML，並部署到 GitHub Pages。

Skills：知道「怎麼做」

適合放 5 步以上且未來會重複做的流程。官方最佳實務建議：如果一個任務超過 5 步且會重複使用，就請 Agent 保存成 skill。

請把剛才「科展前置作業產生流程」存成 skill，名稱為 science-fair-prep。

5.1 教師工作流技能設計範本

Skill 名稱：science-fair-research-prep
用途：把一個科展題目轉成可執行的前置作業文件。
流程：
1. 查詢是否有相似研究與歷屆科展題目。
2. 重新界定題目，避免過度重複。
3. 產生研究動機、研究目的、文獻探討。
4. 設計 3–5 組實驗，每組列變因、材料、步驟、紀錄表。
5. 產出 DOCX/PDF/HTML 所需文字與檔案架構。
6. 最後給老師檢核表與學生分工建議。
不要把所有東西都塞進 Memory。官方文件提到 Memory 容量是 bounded，會合併整理；而且記憶在 session 中寫入後，不會立刻出現在同一個 session 的 system prompt，要到下一個 session 才會反映。

6. Cron 與 Gateway：讓 Hermes 從「問答」變成「例行工作者」

6.1 Cron 能做什麼

Hermes 的 Cron 可用自然語言或 cron expression 建立一次性或週期性任務，也可暫停、恢復、編輯、觸發、刪除任務，並可附加一個或多個 skills；結果可回傳到原聊天、檔案或訊息平台。Cron 每次執行會開新 session、載入指定技能、完成 prompt、交付結果並更新下次執行時間。

/cron add "every day 7am" "整理昨晚 AI 教育新聞，用繁體中文給我 5 點摘要" --skill edu-news-digest
hermes cron create "every Friday 5pm" "檢查 science-portal 專案 README 與網站連結是否需要更新"

6.2 Gateway：讓 Agent 住在你常用的地方

Hermes 支援多種訊息平台與 gateway；官方文件與首頁列出 Telegram、Discord、Slack、WhatsApp、Signal、Email、CLI 等多種介面。實務上，先建議從 Telegram 或 Open WebUI 其中之一開始，確認權限與回覆穩定後再擴充。

6.3 Open WebUI 整合

# ~/.hermes/.env
API_SERVER_ENABLED=true
API_SERVER_KEY=請換成強密碼
hermes gateway
# Open WebUI 連線 URL：
# http://localhost:8642/v1
# Docker 版 Open WebUI 常用：
# http://host.docker.internal:8642/v1
故障排除：如果 Open WebUI 看不到模型，先確認 URL 有包含 /v1，再檢查 curl http://localhost:8642/health 是否回傳 {"status":"ok"}。

7. 高價值應用範例：從教育現場到開發工作流

範例 A｜每日 AI 教育新聞摘要

目標：每天早上自動蒐集 AI 教育、自然科教學、科展相關新聞，傳到 Telegram。

Every morning at 7:30am, search reliable sources for AI education, science fair, and elementary science teaching news. Summarize in Traditional Chinese, include dates, links, and one classroom application idea. Send it to Telegram.

加強版：建立 edu-news-digest skill，規定來源優先序、摘要格式、避免未查證消息。

範例 B｜科展前置研究助理

目標：輸入題目後，自動查重、找文獻、產生研究架構。

請針對「探討光照與濕度對黏液型食蟲植物黏液表現與餌料黏附能力之影響」做科展前置研究。請查找相似題目、提出避免重複的切角，並生成研究動機、目的、實驗設計與紀錄表。

工作流：web search → 整理相似研究 → 產生差異化題目 → 建立 Word/PDF/HTML 初稿。

範例 C｜GitHub Pages 教材網站維護

目標：定期檢查教材網站連結、圖片、RWD、README 與部署狀態。

在目前 repo 中檢查教材網站：
1. 找出所有 HTML、CSS、JS。
2. 檢查 broken links 與圖片路徑。
3. 檢查手機版 RWD 問題。
4. 產生修正 PR 草稿與 changelog。
請不要直接 push，先列出修改計畫。

安全建議：先讓 Agent 做 read-only review，再逐步開放 patch、commit、PR。

範例 D｜研習成果報告自動整理

目標：把活動主題、講師、照片、心得、後續應用整理成成果報告。

請根據我提供的研習資料與照片，產生成果報告：
一、活動基本資料
二、活動目的
三、研習內容與照片說明
四、後續應用與推動方向
五、教師反思
輸出為可轉成 docx/pdf 的結構化內容。

建議：把常用報告格式保存為 skill，下一次只要丟資料即可。

範例 E｜Telegram 團隊助理

目標：讓科展老師群或資訊小組能透過 Telegram 詢問同一個 Agent。

hermes gateway setup
# 設定 Telegram bot token、允許使用者清單、預設 home channel

權限策略：只允許指定 user ID，先關閉危險 shell 指令，敏感 repo 使用唯讀 token。

範例 F｜MCP 連 GitHub Issues

目標：讓 Hermes 只看得到 GitHub issue 管理需要的最小工具，而不是整個 GitHub 權限。

mcp_servers:
  github:
    command: "npx"
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "***"
    tools:
      include: [list_issues, create_issue, update_issue]
      prompts: false
      resources: false

安全重點：使用最小權限、工具白名單、token 分離。

8. 安全設計：把 Agent 當「初階助理」，不要當「全權管理員」

最小權限

訊息平台使用 allowlist；API key 使用專用 token；MCP 僅暴露必要工具。

人類核准

危險命令維持 manual 或 smart approval；不要長期開 --yolo。

沙盒隔離

會執行程式碼或 shell 的任務，優先使用 Docker、SSH、Modal 等隔離後端。

8.1 官方安全層

官方 Security 文件列出多層防護：使用者授權、危險命令核准、容器隔離、MCP credential filtering、context file prompt injection scanning、cross-session isolation、input sanitization。

# ~/.hermes/config.yaml 範例概念
approvals:
  mode: manual
  timeout: 60
  cron_mode: deny
  mcp_reload_confirm: true
  destructive_slash_confirm: true
高風險操作不要自動化：刪除大量檔案、付款、寄送正式信件、公開發文、修改正式資料庫、學生個資處理，都應保留人工確認。

9. Hermes Agent 的極限與風險

極限說明建議做法
安裝與設定仍有門檻雖然官方提供 installer，但 provider、API key、Gateway、MCP、Docker、Cron 仍需要工程判斷。先 CLI 成功，再逐步加功能；使用 hermes doctor 與 --safe-mode 除錯。
模型品質決定上限Hermes 是 Agent 框架，不會讓弱模型突然變成強模型。長鏈任務仍可能出錯。重要任務用高品質模型；讓 Agent 輸出檢核表與引用來源。
工具越多，風險越高MCP、terminal、browser、file patch 一旦開太多權限，錯誤成本上升。按任務開 toolset；敏感工具用白名單；危險命令人工核准。
Memory 不是萬能資料庫官方最佳實務提到 Memory 有容量邊界，且 session 中更新不會立刻進入同一 session 的 prompt。長文件放 repo/doc，不放 memory；流程放 skill；穩定偏好放 memory。
Cron 需要自包含 promptCron job 會在 fresh session 執行，不能假設記得當前聊天細節。排程 prompt 寫清楚來源、格式、交付方式與失敗處理。
自動化可能產生隱性成本長任務、排程、多工具呼叫會消耗 API token 或外部服務成本。設定模型、頻率、預算警示；非 LLM 任務用 no-agent script-only cron。
公開資料仍在快速變動Hermes Agent 是 2026 年仍快速演進的開源專案，指令與平台支援可能變化。以官方 docs/GitHub release 為準；文件定期更新。

10. 建議你實作的 14 天學習路線

第 1–2 天：安裝與基本聊天

完成 Desktop 或 WSL2 安裝，跑 hermes setup --portal 或 hermes model，確認 hermes chat -q 可用。

第 3–4 天：建立個人規則

撰寫 SOUL.md 與一個教材專案的 AGENTS.md，讓 Hermes 知道繁中、RWD、GitHub Pages、教師用語。

第 5–7 天：建立第一個 Skill

選一個常做任務，例如「科展前置作業」或「研習成果報告」，讓 Hermes 保存流程為 skill。

第 8–10 天：串接 Gateway

選 Telegram 或 Open WebUI，先測試只讀任務與摘要任務，不急著開寫入權限。

第 11–12 天：建立 Cron

建立每日教育新聞摘要、每週教材網站檢查、每月科展資料更新任務。

第 13–14 天：安全審查

檢查 token、allowlist、approval、MCP 工具白名單與備份策略，建立「可回滾」流程。
