2. 什麼是沙盒

在程式設計與資安語境中，沙盒是一種「受限制的執行環境」。它讓程式可以執行，但不能任意接觸主機、作業系統、其他程式、使用者資料、網路或敏感資源。

沙盒的核心目的不是讓程式「絕對安全」，而是把風險限制在可控範圍內。

  • 限制程式能看到什麼。
  • 限制程式能讀寫什麼。
  • 限制程式能呼叫哪些系統能力。
  • 限制程式能消耗多少 CPU、記憶體、磁碟與網路。
  • 讓高風險程式即使出錯或被攻擊，也不容易影響整個系統。

可以把沙盒理解為「最小權限原則」的執行環境版本：程式只拿到完成任務所需的最低能力，其餘能力預設不可用。

3. 沙盒扮演的角色

3.1 安全邊界

沙盒最常見的角色是安全邊界。當程式處理不可信輸入、第三方外掛、使用者上傳檔案、網頁內容或 AI 產生的程式碼時，沙盒可避免程式直接操作主機資源。

例如瀏覽器會把網頁渲染、JavaScript 執行、PDF 處理等高風險任務放入受限程序。即使某個網頁觸發了渲染引擎漏洞，攻擊者仍需要突破沙盒限制，才可能進一步控制主機。

3.2 錯誤隔離

沙盒也用於穩定性。若某個插件、頁籤、測試任務或使用者程式崩潰，沙盒能讓錯誤停留在局部，不拖垮主應用程式。

  • 瀏覽器單一頁籤崩潰，不一定造成整個瀏覽器關閉。
  • 線上程式評測系統中，學生程式無限迴圈時，只終止該次執行。
  • CI 測試中，某個測試容器失敗，不污染其他測試環境。

3.3 資源治理

沙盒常搭配配額與限制，例如 CPU 時間、記憶體、檔案系統容量、執行時間、網路連線與程序數量。這類限制可以防止誤用或惡意程式造成拒絕服務。

3.4 可重現環境

沙盒也能提供乾淨、可重現的執行環境。開發、測試、CI/CD 與教學常用容器或虛擬環境，確保依賴版本、檔案狀態與系統設定一致。

4. 沙盒的底層原理

沙盒不是單一技術，而是一組隔離與限制機制。不同場景會採用不同層次的沙盒。

4.1 程序隔離

將不可信任工作放進獨立 process，是常見的第一層隔離。不同 process 原本就有獨立位址空間，不能直接讀寫彼此記憶體。

但僅有 process 隔離通常不夠，因為 process 仍可能呼叫系統 API 存取檔案、網路與裝置。因此實務上會再加上權限限制、系統呼叫過濾或 OS sandbox policy。

4.2 權限與 capability 模型

沙盒會把「可做的事」明確列成授權項目。未授權的能力預設不可用。

  • 可讀取哪些目錄。
  • 可寫入哪些目錄。
  • 是否可以開啟網路。
  • 是否可以存取相機、麥克風、藍牙、USB。
  • 是否可以啟動子程序。
  • 是否可以載入動態函式庫。

4.3 Linux namespaces

Linux namespaces 讓程序看到的系統視圖被隔離。Docker 等容器技術會使用 namespaces 讓容器擁有獨立的 process ID、hostname、mount point、network interface、IPC 資源與 user ID 對應。

4.4 cgroups 資源限制

cgroups 用於限制、統計與控制資源使用量，例如 CPU、memory、I/O 與程序數。Docker 官方文件指出，容器會搭配 namespaces 與 cgroups，前者偏向隔離，後者偏向資源治理。

4.5 seccomp 系統呼叫過濾

seccomp-bpf 可限制程序能呼叫哪些 system call。Linux kernel 文件特別提醒：系統呼叫過濾本身不是完整沙盒，而是降低 kernel attack surface 的一層機制。

實務上 seccomp 常和 namespaces、capabilities、AppArmor、SELinux、chroot 或容器 runtime 一起使用。

4.6 虛擬機與 microVM

如果威脅模型較高，可能會用 VM 或 microVM 作為隔離邊界。相較於容器，VM 有獨立 kernel，隔離通常更強，但啟動、效能與管理成本較高。

4.7 語言或 runtime 層沙盒

某些語言或 runtime 提供受限執行，例如 JavaScript iframe sandbox、Node.js vm、Lua VM、WebAssembly runtime、Python restricted execution 或自訂 import policy。

這類沙盒方便，但不能一概視為強安全邊界。若程式碼完全不可信，通常應優先依賴 OS、容器、VM 或 Wasm runtime 的明確隔離能力。

4.8 WebAssembly / WASI

WebAssembly 的安全模型讓模組在隔離環境中執行，不能直接任意接觸宿主系統。若需要檔案或網路等 I/O，通常要由宿主 runtime 提供 API。

WASI 進一步採用 capability-based 模型。以 Wasmtime 為例，程式只能存取宿主明確授權的檔案或目錄。

5. 常見應用場景

5.1 瀏覽器與網頁內容

瀏覽器每天執行大量不可信網站內容，因此沙盒是瀏覽器安全架構的核心。常見隔離對象包含網頁渲染程序、JavaScript 執行環境、iframe、PDF viewer、擴充套件與 WebAssembly 模組。

常見限制包含同源政策、iframe sandbox 屬性、Content Security Policy、權限提示與分程序架構。

5.2 容器化開發與部署

Docker、Kubernetes 等容器平台常被視為一種沙盒化部署方式。它們提供環境隔離、依賴封裝與資源限制，適合微服務部署、CI/CD 測試、本機開發環境、課堂程式執行環境與批次工作。

容器不是 VM。若容器以 privileged 模式執行、掛載敏感 host path、使用 host network、或暴露 Docker socket，沙盒邊界會大幅削弱。

5.3 線上程式評測與教育平台

執行學生或使用者提交的程式碼時，沙盒非常重要。應限制執行時間、CPU、記憶體、檔案讀寫、網路、子程序數量、系統呼叫與輸出大小。

5.4 外掛與腳本系統

許多應用會允許第三方擴充，例如 IDE 外掛、遊戲 Mod、CMS 外掛、SaaS workflow、使用者自訂公式或 AI agent 工具。這些場景中，良好設計通常是「只暴露必要 API」，而不是讓外掛拿到完整 process 權限。

5.5 行動 App

Android 與 iOS/macOS App Sandbox 都以 OS 層級隔離 App。每個 App 預設只能存取自己的資料，若要使用相機、定位、通訊錄或檔案，需要權限或 capability。

5.6 惡意程式分析

安全研究人員會在沙盒中執行可疑檔案，觀察檔案寫入、系統設定變更、網路連線、程序注入、加密或刪除行為。此類沙盒需要特別防範反沙盒偵測。

5.7 AI 與自動化代理

AI coding agent、資料分析 agent 或工具型 agent 可能會執行模型產生的指令與程式碼。這類輸出不應被視為天然可信。

  • 讀取本機機密檔案。
  • 外傳 API key 或 token。
  • 刪除或覆蓋工作目錄。
  • 執行 package install 時帶入惡意依賴。
  • 被 prompt injection 誘導執行危險命令。

建議使用獨立工作目錄、最小權限 token、網路限制、審計記錄、容器或 microVM 隔離。

6. 常見狀況與風險

6.1 誤以為「有沙盒就安全」

沙盒降低風險，不代表消除風險。任何沙盒都可能有設定錯誤、runtime 漏洞、kernel 漏洞、權限授予過寬、掛載路徑過大、網路出口未限制、日誌或暫存檔洩漏。

設計時應先明確威脅模型：要防的是誤操作、壞掉的程式、惡意使用者，還是高能力攻擊者。

6.2 沙盒逃逸

沙盒逃逸是指程式突破限制，取得沙盒外的資源或更高權限。

  • kernel 或 runtime 漏洞。
  • host 目錄掛載不當。
  • Docker socket 暴露給容器。
  • privileged container。
  • 錯誤的 user namespace 設定。
  • 過寬的 seccomp 或 AppArmor policy。
  • 外掛 API 暴露了危險能力。

6.3 資源耗盡

沙盒內程式可能耗盡資源，即使不能逃逸，也能造成服務不可用。應對方式包含 CPU、memory、pid、file size 限制、timeout、輸出截斷、rate limit、job queue 隔離、配額與清理機制。

6.4 網路與資料外洩

許多沙盒只限制檔案系統，卻允許網路出口。若沙盒內有機密資料，攻擊者可能透過 HTTP、DNS、WebSocket 或套件管理工具外傳。

應預設關閉網路、使用 allowlist、封鎖 metadata service、阻擋內網掃描、隔離 DNS，並對出站流量做審計。

6.5 檔案系統污染

若沙盒共享 host 目錄，程式可能修改原始碼、刪除資料、留下惡意檔案或污染快取。建議使用臨時工作目錄、唯讀資料、輸入輸出分離，以及任務結束後銷毀環境。

6.6 時間、隨機性與可重現性

沙盒可能導致測試結果與本機不同，例如系統時間、timezone、locale、隨機種子、CPU 架構、檔案路徑、權限與網路狀態不同。建議固定 timezone、locale、seed，並記錄 image digest 與 runtime version。

6.8 除錯困難

沙盒會改變環境，導致程式在本機可用、沙盒內失敗。常見原因包含缺少權限、缺少系統套件、無法寫入路徑、DNS 或網路被封鎖、時間限制太短、記憶體限制太小、seccomp 擋掉 system call、AppArmor/SELinux 拒絕存取。

建議保留 sandbox 啟動參數、resource limit、stdout/stderr、kernel audit log、seccomp/AppArmor/SELinux 拒絕紀錄、image digest 與 runtime version。

7. 設計沙盒時的判斷框架

7.1 先定義威脅模型

  1. 執行的程式碼是否完全不可信？
  2. 攻擊者是否能自由提交程式？
  3. 沙盒內是否有機密資料？
  4. 是否允許網路？
  5. 是否需要保護 host、其他租戶或內網？
  6. 失敗時可接受的影響範圍是什麼？

若只是避免測試污染，容器可能足夠。若要執行陌生人的任意程式碼，應考慮強化容器、microVM 或 VM。

7.2 預設 deny

沙盒政策應以 deny by default 為基礎。需要的能力再逐項開放。

  • 預設無網路，需要下載依賴時才給短時間 allowlist。
  • 預設唯讀輸入，只允許寫入 /tmp/output。
  • 預設不掛載憑證，需要部署時使用短效 token。
  • 預設不允許啟動巢狀容器或存取 Docker socket。

7.3 分離控制平面與執行平面

管理任務的系統不應與執行不可信程式碼的環境共用權限。控制平面保存排程、使用者、權限與審計資料；執行平面只拿到單次任務所需資料，任務完成後回收環境。

7.4 資源限制是必要條件

即使不擔心惡意攻擊，也應設定 timeout、memory limit、CPU quota、disk quota、process limit 與 output limit。

7.5 沙盒不能取代輸入驗證

沙盒是隔離層，不是輸入清理器。Web 應用仍需驗證輸入、輸出編碼、HTML sanitizer、Content Security Policy，並避免不必要的 eval 或動態執行。

8. 實務檢查表

8.1 基本安全檢查

  • 是否明確定義不可信輸入與不可信程式碼？
  • 是否使用最小權限？
  • 是否限制檔案讀寫範圍？
  • 是否預設禁止或限制網路？
  • 是否設定 CPU、memory、pid、disk、timeout？
  • 是否避免把 host secret 掛入沙盒？
  • 是否避免 privileged container？
  • 是否避免掛載 Docker socket？
  • 是否使用唯讀 root filesystem？
  • 是否有審計紀錄？
  • 是否定期更新 runtime、kernel 與 base image？

8.2 容器沙盒檢查

  • 啟用 user namespace 或 rootless 模式。
  • 限制 Linux capabilities。
  • 啟用 seccomp profile。
  • 啟用 AppArmor 或 SELinux。
  • 使用 non-root user。
  • 不使用 --privileged。
  • 不使用 host network，除非有明確理由。
  • 不掛載 /、/var/run/docker.sock、cloud credentials。
  • 對 volume 掛載使用唯讀或最小範圍。

8.3 執行使用者程式碼檢查

  • 每次任務使用乾淨環境。
  • 測資與答案分離。
  • 禁止或限制網路。
  • 限制輸出大小。
  • 任務完成後銷毀環境。
  • 對編譯器、package manager、interpreter 做版本鎖定。
  • 記錄執行命令、限制、結果與錯誤。

9. 小結

沙盒在程式設計中是一種風險邊界：它允許程式執行，但限制程式能接觸的資源與能力。它可用於安全防護、錯誤隔離、資源治理與可重現環境。

沙盒的實作可能位於語言 runtime、瀏覽器、作業系統、容器、WebAssembly、VM 或 microVM 層。隔離層越接近 OS 或硬體，通常安全邊界越強，但成本也越高。

實務上最重要的觀念是：沙盒不是單一開關，而是一組策略。好的沙盒設計需要明確威脅模型、最小權限、資源限制、網路控制、審計紀錄與定期更新。對於完全不可信的程式碼，不能只依賴語言層限制或預設容器設定，應採用更完整的多層防護。
