🐻 黑熊老師科展日誌

完整開發與部署流程紀錄

網站：https://science-exhibition-production.up.railway.app　·　原始碼：github.com/prayer168/Science-Exhibition

一、專案簡介

本專案是一個「科展研究日誌」網站，靈感與欄位設計取自《科展研究日誌》學生版 Word 檔。目標是讓學生／老師能線上記錄科展研究的每一步，並對外展示研究歷程。

網站分為兩大部分：對外公開的「日誌牆」前台，以及需 Google 登入的「管理後台」。核心記錄欄位為：年月日、當天科展所做的事情、遇到的問題、待辦事項；另含時間、地點、記錄者、發現與數據、照片等輔助欄位。

二、技術選型

項目採用技術原因
前端 / 後端框架Next.js 16（App Router）+ TypeScript前後端一體、檔案式路由、部署簡單
樣式Tailwind CSS v4快速打造一致的熊主題介面
資料庫SQLite（better-sqlite3）輕量、零設定、適合單機/小型應用
登入驗證NextAuth v5（Auth.js）整合 Google OAuth，以 email 清單控管
Word 匯出docx程式化產生 .docx，重現日誌表格
PDF 匯出puppeteer用 Chromium 把 HTML 轉成 PDF，中文字型完美
部署平台Railway（Docker + Volume）有持久磁碟，SQLite/照片/puppeteer 幾乎原樣可跑

三、開發歷程

階段一：需求分析

先讀取使用者提供的 Word 日誌檔（科展研究日誌學生版），解析其內容與每次記錄的欄位結構，據此設計資料庫欄位與頁面。

  • 解析 .docx，擷取「科展研究日誌 第 N 次」的格式：日期、時間、地點、記錄者，以及上課/研究內容、發現與數據、問題與待辦、下次改進等欄位。
  • 確認核心四欄位：年月日、當天所做的事、遇到的問題、待辦事項。

階段二：建立基礎網站

用 create-next-app 建立專案，安裝 better-sqlite3，建立資料層與 API，再做前台日誌牆與後台 CRUD。

  1. 建立 Next.js 專案（TypeScript + Tailwind + App Router）。
  2. lib/db.ts：建立 SQLite 連線、journals 資料表與 CRUD 函式。
  3. API 路由：/api/journals（列表/新增）、/api/journals/[id]（單筆/更新/刪除）。
  4. 前台 app/page.tsx：卡片式日誌牆，依日期排序，問題用紅色、待辦用藍色標示。
  5. 後台 app/admin：日誌列表表格、新增頁、編輯頁、共用表單與刪除按鈕。
  6. 設定 next.config.ts 將 better-sqlite3 列為 serverExternalPackages，確保原生模組正常載入。
💡 以瀏覽器實測前台、後台、新增表單，並確認中文 UTF-8 顯示正確（PowerShell 終端顯示的亂碼只是編碼問題，資料庫存的是正確的 UTF-8）。

階段三：四大進階功能

1. Google 帳號登入

  • 用 NextAuth v5 加入 Google Provider，建立 auth.ts、/api/auth/[...nextauth] 路由。
  • middleware.ts 保護 /admin 頁面與所有寫入型 API（非 GET）。
  • 以 ALLOWED_EMAILS 環境變數作為允許清單，只有清單內的帳號能進後台。
  • 自動產生 AUTH_SECRET，並建立 .env.local 範本與設定說明。

2. 搜尋與篩選

  • 資料層 getAll 支援關鍵字與日期區間查詢。
  • API 與前/後台都能依關鍵字（內容/問題/待辦/地點/記錄者）與起訖日期過濾。
  • 新增共用 SearchBar 元件。

3. 照片拖拉與多筆上傳

  • 新增 photos 資料表與上傳/刪除 API（多檔上傳）。
  • PhotoUploader 元件：可拖拉多張圖片或點擊多選，含即時預覽與刪除。
  • 新增時暫存於前端、儲存後一併上傳；編輯時即時管理既有照片。

4. 匯出 Word / PDF / HTML

  • 建立共用匯出資料層（含照片以 base64 內嵌），與一套日誌表格 HTML 模板。
  • HTML：直接輸出自給自足的檔案；Word：用 docx 重現表格；PDF：用 puppeteer 轉 HTML。
  • 單篇或全部皆可匯出，忠實重現原始日誌表格格式（第 N 次 / 日期時間地點記錄者 / 各欄位 / 照片）。
💡 以直接呼叫程式庫的測試腳本驗證匯出：HTML/DOCX/PDF 全部產出成功，PDF 經檢視確認版面與中文字型正確。

階段四：登入體驗優化

將原本的「管理後台登入」中間頁拿掉，改成點「登入」或進入後台時直接跳轉 Google 登入畫面；/login 改為自動觸發，並保留被拒絕時的錯誤提示以避免無限跳轉。

階段五：部署到 GitHub

  1. 初始化 git、設定身分、確認 .env.local（含密鑰）被 .gitignore 排除。
  2. 加上 !.env.local.example 例外，讓不含機密的範本可進版控。
  3. 安裝 GitHub CLI（gh），由使用者完成 gh auth login 授權。
  4. 建立公開 repo 並 push：gh repo create Science-Exhibition --public --source=. --push。

階段六：部署到 Railway（正式上線）

因原本為「本機/持久磁碟」設計，serverless 平台會有資料庫、上傳、PDF 三個不相容點，故選擇有持久磁碟的 Railway，並用 Docker 容器化。

程式調整

  • 資料路徑改用 DATA_DIR 環境變數（Railway 上指向 /data 的 Volume）；DB 與照片都存這裡。
  • 照片不再放 public/，改由 /api/uploads/[name] 路由提供（含路徑穿越防護），容器重啟也不遺失。
  • puppeteer 改用容器內系統 chromium（PUPPETEER_EXECUTABLE_PATH）。
  • 建立 Dockerfile：Node 22 + chromium + 思源黑體（fonts-noto-cjk）+ better-sqlite3 編譯工具。
  • auth.ts 加上 trustHost: true，讓 NextAuth 在反向代理後方正常運作。

Railway 設定步驟

  1. New Project → Deploy from GitHub repo → 授權並選 Science-Exhibition（自動偵測 Dockerfile 開始建置）。
  2. 右鍵服務卡片 → Attach Volume，Mount path 設為 /data。
  3. Variables 設定：DATA_DIR、AUTH_URL、AUTH_SECRET、AUTH_GOOGLE_ID、AUTH_GOOGLE_SECRET、ALLOWED_EMAILS。
  4. Settings → Networking → Generate Domain 產生公開網址。
  5. Google Cloud Console 建立 OAuth 用戶端，重新導向 URI 設為「網址 + /api/auth/callback/google」，並加測試使用者。

疑難排解（實際遇到並解決）

問題原因解法
502 Bad GatewayRailway 注入 PORT=8080，網域卻指向 3000把公開網域的 target port 改成 8080（與 app 實際監聽一致）
登入出現「已封鎖存取權／invalid_request」AUTH_URL 被填壞，混入 localhost（錯誤頁出現 localhohttps）把 AUTH_URL 改成正確的 https 正式網址後重新部署
Google 擋下登入帳號不在 OAuth 測試使用者清單於 OAuth 同意畫面（對象）新增 prayer168@gmail.com 為測試使用者

四、最終成果

  • 公開日誌牆 + 需登入管理後台。
  • 核心欄位：年月日、當天所做的事、遇到的問題、待辦事項（+ 發現、照片等）。
  • Google 登入（email 允許清單控管）。
  • 關鍵字 / 日期搜尋與篩選。
  • 照片拖拉、多筆上傳。
  • 匯出 Word / PDF / HTML，重現原始日誌表格。
  • 已部署上線並文件化維護流程。

五、日常維護流程

核心原則：把改動 push 到 GitHub，Railway 就會自動重新部署。

  1. 本機改程式碼（cd science-exhibition，編輯檔案）。
  2. 本機測試：npm run dev（http://localhost:3000，測完 Ctrl+C）。
  3. 推上 GitHub：git add -A → git commit -m "說明" → git push。
  4. Railway 自動部署：於 Deployments 看進度，約 2–5 分鐘變 Active，開網址確認。
想做什麼怎麼做
改功能 / 版面改程式 → git push → 等 Railway 部署
改環境變數（密鑰、允許 email）Railway → Variables 直接改（自動重部署，不用 push）
新增可登入的人ALLOWED_EMAILS 加 email；並到 Google 測試使用者加同一個 email
改壞了要還原Railway → Deployments → 選正常舊版 → ⋮ → Redeploy
看錯誤Railway → Deployments → 最新部署 → Deploy / Build Logs
⚠️ AUTH_URL 一定要是正確的正式網址（打錯登入會壞）；不要刪除 /data Volume，否則所有日誌與照片會消失；.env.local 永不進版控，正式密鑰只存在 Railway。
