Antigravity IDE 與 Antigravity 2.0

差異比較、正確使用方式、安全設定與教材網站開發建議

核心概念差異比較如何選擇使用流程安全設定提示詞範例教師應用

一、核心概念

Antigravity 相關工具可以分成兩種使用思維：一種是像傳統 IDE 一樣直接編輯程式；另一種則是把 AI Agents 當成專案助理，讓它們協助規劃、修改、測試與回報。

編輯器導向Antigravity IDE

比較像 VS Code、Cursor 這類 AI 輔助程式編輯器，適合直接開啟專案資料夾，修改 HTML、CSS、JavaScript、Python 等檔案。

Agent 導向Antigravity 2.0

比較像 AI 專案任務控制台，適合把任務交給多個 AI Agents，讓它們進行規劃、分工、執行與成果回報。

簡單判斷：你要自己看著程式碼改，用 Antigravity IDE；你要分派任務給 AI Agents，使用 Antigravity 2.0。

三、如何正確選擇工具

日常小修改

例如改教材網頁文字、調整按鈕、修正圖片路徑、檢查 CSS。建議使用 Antigravity IDE。

中型網站開發

例如建立一份可以部署到 GitHub Pages 的互動教材網站。建議以 IDE 為主，2.0 為輔。

大型專案管理

例如同時要做網頁、README、測試、部署說明與多頁教材。建議使用 Antigravity 2.0。

四、建議使用流程

流程 A：使用 Antigravity IDE

開啟專案資料夾：使用 Open Folder 選擇教材或網站專案。
先要求分析：請 AI 先閱讀專案架構，不要立即修改檔案。
確認修改範圍：要求 AI 列出會動到哪些檔案。
分階段修改：一次只改一個問題，例如 RWD、圖片、互動功能或部署設定。
測試結果：請 AI 說明如何在瀏覽器中測試，並確認 GitHub Pages 路徑是否正確。

流程 B：使用 Antigravity 2.0

定義總任務：例如建立完整教材網站、重構專案或補齊文件。
要求 Agent 先提出計畫：不要一開始就讓 Agent 直接修改。
拆分任務：可分成介面設計、功能開發、測試、文件、部署。
審查 Artifacts：查看 Agent 產出的計畫、截圖、測試紀錄與修改摘要。
逐步核准：每完成一階段再進入下一階段，避免一次改太多造成錯誤。
最佳做法：對教師教材開發而言，建議先用 Antigravity IDE 做小範圍修改，等專案變大或需要多任務分工時，再使用 Antigravity 2.0。

五、安全設定建議

AI Agent 工具可能會讀取專案檔案、執行終端機指令、修改程式碼或操作瀏覽器。初學者不要一開始就開啟完全自動化權限。

設定項目建議設定原因
Terminal Execution PolicyRequest Review執行終端機指令前先讓使用者確認
Review PolicyRequest Review 或 Agent Decides初學建議先採人工審核
File Access只允許目前 workspace避免 AI 讀取或修改非專案資料
Browser Allowlist只允許可信任網站降低瀏覽器操作與網頁提示注入風險
自動執行高風險指令不要開啟避免誤刪檔案、覆蓋資料或安裝不必要套件
特別提醒：不要讓 AI 自動執行刪除、格式化、清空資料夾、重新初始化 Git repository 等高風險指令。

六、實用提示詞範例

提示詞 1：先分析、不修改
請先不要修改任何檔案。
請先閱讀目前專案架構，列出：
1. 主要檔案功能
2. 目前可能的問題
3. RWD 是否完整
4. 是否適合部署到 GitHub Pages
5. 建議修改順序

等我確認後，再開始修改。
提示詞 2：安全修改規則
請依照剛才的建議修改。
限制：
1. 不要刪除既有功能
2. 修改前先列出會動到哪些檔案
3. 每次只完成一個階段
4. 完成後請提供測試方式
5. 不要執行會刪除檔案的指令
提示詞 3：教材網站檢查
請檢查這個教材網站是否適合部署到 GitHub Pages。
請特別檢查：
1. index.html 是否在正確位置
2. 圖片與 CSS、JavaScript 路徑是否正確
3. 手機與平板版面是否正常
4. 是否有無法載入的資源
5. 是否需要補 README.md

請先列出問題，不要直接修改。
提示詞 4：Antigravity 2.0 多 Agent 任務
請協助我規劃這個教材網站專案。
請將任務拆成以下幾個部分：
1. 專案架構檢查
2. RWD 版面優化
3. JavaScript 互動功能測試
4. GitHub Pages 部署檢查
5. README 與教學說明文件整理

請每個任務都先提出計畫，等我確認後再執行。

七、給教師教材開發的建議

如果主要目標是製作自然科互動教材、RWD 網頁、課程說明頁、GitHub Pages 教材平台，建議採用「IDE 先行、2.0 輔助」的方式。

平常備課與小型教材

使用 Antigravity IDE。每次只處理一個頁面或一個功能，避免一次修改太多造成錯誤。

大型教材平台或專案成果

使用 Antigravity 2.0。先請 Agent 拆解工作，再逐步確認設計、程式、測試與文件。

建議標準流程

  1. 先備份原始專案。
  2. 用 Antigravity IDE 檢查單一功能。
  3. 確認可正常運作後，再進行下一階段。
  4. 大型專案再交給 Antigravity 2.0 拆任務。
  5. 每次修改後都測試手機、平板與桌機版面。
  6. 部署到 GitHub Pages 前，檢查檔案路徑與 README。
結論：Antigravity IDE 適合「看著程式碼做精準修改」；Antigravity 2.0 適合「把大型任務交給 AI Agents 分工處理」。對教材開發者來說，兩者不是互相取代，而是依照任務大小搭配使用。
