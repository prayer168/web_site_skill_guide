1. RAG 檢索摘要

「Rolling AI」不是常見的正式技術名詞。本文將它整理為 vibe coding 中的工程化實作方法，並參考三個相近脈絡：vibe coding 的自然語言協作、AI agent 的觀察與修正迴圈、以及 rolling update / rolling window 的小步滾動概念。

2. 操作性定義

在本文中，Rolling AI 指的是：在 vibe coding 裡，用一輪一輪可檢查的小步驟，讓 AI 持續根據最新狀態、測試結果與人類回饋修正程式。

它不是單純「一直叫 AI 寫程式」，而是把 AI 協作設計成有節奏的閉環：

  • 每輪只處理一個明確目標。
  • 每輪都要讀取目前程式狀態。
  • 每輪都要產出可執行、可測試的結果。
  • 每輪結束後依據結果修正下一輪指令。
  • 必要時暫停、回滾或縮小範圍。

3. 核心流程

Rolling AI 可以用以下流程描述：

  1. Observe：先讀現有程式、錯誤訊息、畫面或測試結果。
  2. Scope：把下一步限制在一個小任務。
  3. Prompt：給 AI 明確的修改目標、限制與驗收標準。
  4. Act：讓 AI 修改程式或產出設計。
  5. Verify：執行測試、開頁面、看截圖或檢查輸出。
  6. Feedback：把具體結果回饋給 AI，進入下一輪。

4. 在 Vibe Coding 裡如何使用

4.1 先定義最小可跑版本

不要一開始要求 AI 建完整產品。先要求一個可以打開、可以執行、可以驗證的最小版本。

範例：先做登入畫面，不做資料庫；先做計時器，不做會員與統計；先做一頁技術文件，不做整站 CMS。

4.2 每輪只處理一個變更

一輪只改一個功能、修一個 bug、補一個版面或加一個測試。這可以讓錯誤容易定位，也能避免 AI 大量重寫。

4.3 用具體觀察取代抽象感覺

不要只說「不好看」或「壞掉了」。要說明在哪個頁面、哪個按鈕、哪個螢幕寬度、哪個錯誤訊息。

4.4 每輪結束要驗收

驗收可以是人工操作，也可以是自動測試、靜態檢查、截圖檢查或部署後 HTTP 檢查。

5. 提示範例

5.1 起始輪

請先做一個單頁 HTML，主題是 Rolling AI。包含標題、目錄、核心流程、使用方法。先不要加動畫或部署。

5.2 功能輪

現在加入三個 SVG 動畫：AI 協作迴圈、提示回饋流程、風險閘門。請保持單一 HTML，不使用外部素材。

5.3 修正輪

手機版表格太寬。請只修 RWD，保留桌機版視覺，不要改文章內容。

5.4 驗收輪

請檢查公開網址是否回傳 200，並確認頁面包含 title、h1、三個 SVG 動畫與 prefers-reduced-motion。

6. 品質控制

Rolling AI 的價值在於把 vibe coding 從「憑感覺接受 AI 結果」改成「小步驗證」。常見控制點如下：

控制點 目的 實作方式
範圍限制 避免一次改太多 每輪指定只改某個檔案、功能或 bug。
差異檢查 知道 AI 改了什麼 看 git diff、檔案列表與變更摘要。
自動測試 避免已知功能退化 執行 unit test、lint、build、型別檢查。
人工驗收 確認體驗符合需求 開頁面、操作流程、檢查 RWD 與文案。
回滾點 保留退路 小 commit、分支、部署前確認。

7. 適用場景

  • 原型設計：快速把想法做成可操作畫面。
  • 教學網頁：逐步加入內容、互動、RWD、動畫與部署。
  • 小型工具：先完成核心流程，再補資料保存、匯出與設定。
  • 舊專案修補：每次只修一個 bug 或一段 UI。
  • 文件與範例：先建立架構，再補圖解、程式碼與檢查表。

8. 風險與限制

8.1 上下文漂移

AI 可能忘記前面約束，或把任務擴大。應在提示中重申「只改這個範圍」，並用 diff 檢查。

8.2 無意識接受錯誤

Vibe coding 容易讓使用者只看表面效果。對資料、安全、金流、權限、部署設定等高風險部分，必須增加人工審查與測試。

8.3 技術債累積

每輪只求能跑，容易讓結構變亂。建議定期安排「整理輪」，只處理命名、重複程式碼、模組邊界與測試。

8.4 不適合高風險一次上線

醫療、金融、法律、資安、身分權限等高風險系統，不應只靠 vibe coding 與 Rolling AI 上線，需完整工程流程與責任審查。

9. 實務檢查表

9.1 每輪開始前

  • 這一輪是否只有一個明確目標？
  • 是否指定不要修改的範圍？
  • 是否提供目前錯誤、截圖、測試結果或檔案位置？
  • 是否定義完成條件？

9.2 每輪完成後

  • 是否看過變更檔案與 diff？
  • 是否執行必要測試或手動操作？
  • 是否確認沒有無關重寫？
  • 是否留下可回滾的 commit 或版本點？

9.3 進入下一輪前

  • 是否把驗證結果轉成具體回饋？
  • 是否縮小下一輪任務？
  • 是否需要先整理程式，而不是繼續加功能？
  • 是否需要人工審查安全或資料風險？
