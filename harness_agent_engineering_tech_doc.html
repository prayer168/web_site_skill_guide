🤖 AI Agent 技術文件｜Harness Agent × Harness Engineering

讓 AI Agent 不只是聰明，而是可控、可驗證、可交付

Harness Engineering 指的是替 AI Agent 建立可運行的工程環境：任務規格、上下文、工具、權限、測試、回饋迴圈與人工審核。它不是單一 Prompt 技巧，而是一套讓 Agent 穩定工作的「駕馭系統」。

01｜概念定位

Harness 是什麼？為什麼最近大家開始談 Harness Engineering？

Harness 原本有「繫具、駕馭、控制並利用力量」的意思。在 AI Agent 場景中，它被用來描述「模型以外的一整套工程環境」。Martin Fowler 網站上的文章將它概括為：在 Agent 裡，Model 之外的部分都可被視為 Harness；也就是讓 Agent 能看見資訊、呼叫工具、接受限制、取得回饋並完成任務的外部系統。

OpenAI 在〈Harness engineering: leveraging Codex in an agent-first world〉中也強調，當 Coding Agent 進入真實軟體開發時，困難不只在模型能力，而在於如何設計環境、回饋迴圈與控制系統，避免文件腐化、規則失效、測試不足與任務漂移。

一句話：Prompt Engineering 是「怎麼問 AI」；Context Engineering 是「給 AI 看什麼」；Harness Engineering 是「建立什麼系統，讓 AI 能安全完成工作」。
Prompt

提示層

設計任務描述、角色、輸出格式、限制條件。適合單次或短任務。

Context

上下文層

整理規格、文件、程式碼、歷史紀錄、RAG 與記憶，讓模型看到正確資訊。

Harness

系統層

整合工具、權限、測試、審核、CI、回滾、觀測與人機協作流程。

02｜Harness Agent

什麼是 Harness Agent？

定義

Harness Agent 可以理解為「被 Harness 駕馭的 AI Agent」。它不是單純聊天機器人，而是能在特定工程環境中執行任務的代理：讀取專案、修改檔案、執行測試、產生報告、開 PR、根據錯誤自我修正，並在必要時交給人審核。

典型能力

讀取上下文呼叫工具執行測試自我修正產生變更紀錄請求人類確認

常見例子包含 Coding Agent、文件 Agent、測試 Agent、資料整理 Agent、教學教材產生 Agent。

Harness Agent 的基本架構

使用者任務
  ↓
任務規格 Task Spec：目標、限制、完成標準
  ↓
Context Layer：專案文件、程式碼、API、範例、記憶
  ↓
Tool Layer：檔案系統、Git、測試工具、瀏覽器、資料庫、MCP
  ↓
Policy / Guardrail：權限、禁止區域、人工確認點
  ↓
Execution Loop：計畫 → 行動 → 測試 → 修正 → 摘要
  ↓
Sensor Layer：linter、type checker、unit test、e2e test、AI review
  ↓
Human Review：PR、變更摘要、風險說明、合併或退回

03｜Harness Engineering

Harness Engineering 的工程目標

目標做法產出
讓 Agent 不偏題任務規格、完成定義、禁止事項、範例輸出可追蹤的任務卡與驗收標準
讓 Agent 看對資料AGENTS.md、README、架構文件、RAG、文件索引乾淨且可更新的上下文系統
讓 Agent 用對工具工具白名單、MCP Server、CLI、測試指令受控工具介面與操作紀錄
讓 Agent 犯錯會被發現lint、type check、unit test、e2e test、coverage、AI review自動化回饋與失敗原因
讓人類可以接手變更摘要、風險標記、PR、rollback plan可審查、可回滾、可維護的交付流程

04｜Guides × Sensors

建立 Harness 的核心：前饋引導與回饋感測

Martin Fowler 網站上的 Harness Engineering 文章將 Coding Agent 的 Harness 分析為多種引導與感測機制。簡化來說，Guides 是 Agent 行動前的引導，Sensors 是 Agent 行動後的回饋；兩者都可以是確定性的工具，也可以是推理型的 AI 判斷。

Computational｜確定性
Inferential｜推理性
Guide 前饋
型別系統、API schema、專案規範、固定指令、禁止清單
AGENTS.md、任務規劃、AI 產生設計草案、語意化規則
Sensor 回饋
ESLint、Prettier、TypeScript、pytest、Playwright、CI
AI code review、架構審查、測試覆蓋建議、風險摘要

05｜建置流程

如何從零開始建置一個 Harness Agent 工作流？

定義任務邊界

先決定 Agent 可以做什麼、不能做什麼。例如：只能修改 src/ 與 docs/，不能改部署金鑰、不能直接推送 main。

建立 AGENTS.md 或專案操作手冊

放入專案結構、常用指令、測試方法、程式風格、輸出格式、PR 規範與安全限制。避免寫成一大坨無法維護的長文件，最好分層、分檔、可檢查。

準備工具層

給 Agent 明確的工具：Git、套件管理器、測試指令、瀏覽器測試、文件產生器、MCP Server。每個工具都要有權限限制與錯誤回報。

設計感測器

先放確定性工具：lint、format、type check、unit test；再加入推理型檢查：AI review、安全風險摘要、可讀性檢查。

設計 Steering Loop

讓 Agent 在測試失敗時讀取錯誤、修正、再測一次；若失敗超過次數或碰到高風險檔案，就停止並請人類介入。

建立人工審核與回滾

所有重要變更都要有摘要、風險、測試結果與 rollback plan。Agent 可以產生 PR，但合併權限最好保留給人。

07｜風險控管

Harness Agent 常見風險與防護

任務漂移

Agent 做了額外修改。防護：限制可修改範圍、要求先列計畫、超範圍即停止。

文件腐化

AGENTS.md 過時。防護：文件分層、指定 owner、定期清理與測試文件有效性。

測試假通過

Agent 改測試而非改問題。防護：禁止任意降低測試標準，變更測試需額外標記。

權限過大

Agent 可刪檔或部署。防護：沙盒、最小權限、敏感操作人工確認。

安全資料外洩

把金鑰或學生資料放入上下文。防護：遮蔽個資、排除 .env、使用資料分級。

無法追蹤

不知道 Agent 改了什麼。防護：Git diff、變更摘要、測試紀錄、PR 審查。
