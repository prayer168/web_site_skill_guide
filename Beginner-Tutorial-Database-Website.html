🐻 從零打造一個會員制資料庫網站

新手友善開發教學 — 以「黑熊老師科展日誌」為例

適合對象：剛開始 vibe coding、第一次接觸後端網頁與資料庫的初學者

寫在前面：這份教學要帶你做什麼

我們會從零做出一個真正能上線的網站：對外有一面「日誌牆」展示資料，對內有需要登入的「管理後台」可以新增、編輯、刪除、搜尋、上傳照片，還能把資料匯出成 Word / PDF / HTML。最後把它部署到網路上，任何人都能用網址打開。

這份文件不只是「我們做了什麼」，而是「你也能照著做」。每個步驟都有觀念解說與可複製的程式碼範例。看不懂的名詞，文末有「名詞速查表」。

💡 學習建議：不要一次讀完。先讀「觀念篇」建立地圖，再跟著「實作篇」一步步做，做到哪卡住就回來查。動手做一次，勝過讀十遍。

Part 1　觀念篇：先看懂整張地圖

1-1　什麼是「vibe coding」？

Vibe coding 指的是：你用自然語言（中文也行）描述想要什麼，讓 AI 幫你寫程式，你負責「指揮、檢查、決定方向」。你不必背語法，但你需要懂「在做什麼、為什麼這樣做」，才能判斷 AI 給的東西對不對。

可以把自己想成「導演」，AI 是「很強但需要明確指示的工程師」。導演不必會操作每台攝影機，但要清楚知道想要的畫面，並在拍壞時看得出來。

  • 你做的事：描述需求、提供範例、檢查結果、做決定、要求修正。
  • AI 做的事：寫程式碼、解釋原理、找錯、提供選項。
  • 關鍵能力：把模糊的想法講清楚（明確的指令），以及驗證結果（不要盲目相信）。

1-2　前端 vs 後端，用餐廳比喻

一個網站通常分成「前端」和「後端」兩部分，想像成一間餐廳：

角色餐廳比喻在網站裡負責
前端 Frontend外場、菜單、餐桌使用者看得到、點得到的畫面（按鈕、表單、文字）
後端 Backend廚房看不到的運算：接收請求、處理邏輯、決定回傳什麼
資料庫 Database倉庫、冰箱長期保存資料的地方（日誌、照片資訊、帳號）
API服務生在前端與後端之間傳話：把點單送進廚房，把菜端出來

當你在網頁按下「新增日誌」：前端（外場）把你填的資料交給 API（服務生），送進後端（廚房）處理，後端把資料存進資料庫（倉庫），再回報「成功」，前端就更新畫面。整個過程就是這樣來回。

1-3　什麼是資料庫？為什麼不能只用變數？

程式裡的變數，網頁一重新整理或伺服器一重開就消失了。資料庫則是把資料「寫到硬碟長期保存」，關機再開也還在。可以把它想成一個超強的試算表（Excel）：有很多「資料表（table）」，每張表有「欄位（column）」與一筆筆「資料列（row）」。

我們用的是 SQLite —— 一種輕量、不需要另外架伺服器的資料庫，整個資料庫就是一個檔案。很適合初學者與中小型網站。

操作資料庫的語言叫 SQL。最常用的四個動作合稱 CRUD：

英文中文SQL 關鍵字白話
Create新增INSERT加一筆新資料
Read讀取SELECT查出資料
Update更新UPDATE改某筆資料
Delete刪除DELETE刪掉某筆資料

1-4　一筆資料的旅程（請記住這張流程）

這是整個後端網站最核心的概念，務必記住這條「資料流」：

使用者在瀏覽器操作（前端）
      │  例如：填表後按「送出」
      ▼
  發出請求 (HTTP request) 給 API
      │  例如：POST /api/journals
      ▼
  後端的「路由處理函式」接收
      │  驗證資料、決定怎麼做
      ▼
  對「資料庫」下 SQL 指令
      │  例如：INSERT INTO journals ...
      ▼
  資料庫存好，回傳結果
      │
      ▼
  後端回應 (HTTP response) 給前端
      │  例如：{ "id": 1, ... } 與狀態碼 201
      ▼
  前端拿到結果，更新畫面給使用者看

之後不管功能多複雜，本質都是這條流程的重複與變化。卡住時，回來想：現在是卡在哪一段？

Part 4　設計與串接資料庫（重點章節）

4-1　先想清楚要存什麼

動手前先列出「要記錄哪些欄位」。我們的日誌每筆需要：日期、做了什麼、遇到的問題、待辦事項，再加上時間、地點、記錄者、發現、照片編號等。把它對應成資料表的欄位：

欄位名型別說明
idINTEGER（主鍵，自動編號）每筆資料的唯一編號
dateTEXT年月日（必填）
contentTEXT當天科展所做的事情（必填）
problemsTEXT遇到的問題
todosTEXT待辦事項
created_atTEXT建立時間（自動）

4-2　用 SQL 建立資料表

CREATE TABLE 就是「開一張新表並定義欄位」。IF NOT EXISTS 代表「如果還沒有才建立」，避免每次啟動都報錯。

CREATE TABLE IF NOT EXISTS journals (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,  -- 自動編號的主鍵
  date        TEXT NOT NULL,    -- 必填
  content     TEXT NOT NULL,    -- 必填
  problems    TEXT,             -- 可空白
  todos       TEXT,             -- 可空白
  created_at  TEXT DEFAULT (datetime('now','localtime'))  -- 預設帶入現在時間
);

4-3　在程式裡連線資料庫（lib/db.ts）

這支檔案做三件事：開啟資料庫檔、建立資料表、提供 CRUD 函式給其他程式呼叫。以下是精簡版，每段都有註解：

import Database from "better-sqlite3";
import path from "path";
import fs from "fs";

// 1) 決定資料庫檔放哪。用環境變數 DATA_DIR，沒設就放專案的 data/ 資料夾
const DIR = process.env.DATA_DIR || path.join(process.cwd(), "data");
if (!fs.existsSync(DIR)) fs.mkdirSync(DIR, { recursive: true }); // 沒有就建資料夾
const db = new Database(path.join(DIR, "journals.db"));          // 開啟（或建立）資料庫檔

// 2) 啟動時確保資料表存在
db.exec(`
  CREATE TABLE IF NOT EXISTS journals (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    date TEXT NOT NULL,
    content TEXT NOT NULL,
    problems TEXT,
    todos TEXT,
    created_at TEXT DEFAULT (datetime('now','localtime'))
  );
`);

// 3) 提供 CRUD 函式
export const journalDb = {
  // 讀取全部（依日期新到舊）
  getAll() {
    return db.prepare("SELECT * FROM journals ORDER BY date DESC").all();
  },
  // 讀取單筆
  getById(id) {
    return db.prepare("SELECT * FROM journals WHERE id = ?").get(id);
  },
  // 新增（用 @name 對應物件欄位，避免 SQL injection）
  create(data) {
    const r = db.prepare(
      "INSERT INTO journals (date, content, problems, todos) VALUES (@date,@content,@problems,@todos)"
    ).run(data);
    return this.getById(r.lastInsertRowid);
  },
  // 刪除
  delete(id) {
    db.prepare("DELETE FROM journals WHERE id = ?").run(id);
  },
};
⚠️ 重要安全觀念：永遠不要用字串拼接把使用者輸入直接組進 SQL（例如 "...WHERE id=" + id）。要用「?」或「@name」這種參數化寫法，才能防止 SQL injection（惡意輸入竄改你的查詢）。
Next.js 需要在 next.config.ts 把 better-sqlite3 列為 serverExternalPackages，原生模組才能正常載入。這類設定細節，正是請 AI 幫你處理的好時機。

Part 5　寫後端 API（讓前端能存取資料）

在 App Router 裡，建立 app/api/journals/route.ts，匯出 GET、POST 等函式，就成了一個 API 端點。GET 用來查資料、POST 用來新增。

import { NextRequest, NextResponse } from "next/server";
import { journalDb } from "@/lib/db";

// GET /api/journals  → 回傳所有日誌（JSON）
export async function GET() {
  const journals = journalDb.getAll();
  return NextResponse.json(journals);
}

// POST /api/journals → 新增一筆
export async function POST(request: NextRequest) {
  const body = await request.json();        // 取出前端送來的資料

  // 後端驗證：必填欄位要有
  if (!body.date || !body.content) {
    return NextResponse.json({ error: "日期和內容為必填" }, { status: 400 });
  }

  const journal = journalDb.create({
    date: body.date,
    content: body.content,
    problems: body.problems ?? null,  // 沒給就存 null
    todos: body.todos ?? null,
  });
  return NextResponse.json(journal, { status: 201 }); // 201 = 已建立
}

5-1　動態路由：針對某一筆資料

資料夾命名成 [id]，就能接到網址裡的編號，例如 /api/journals/5。用來查單筆、更新、刪除。

// 檔案：app/api/journals/[id]/route.ts
export async function DELETE(_: NextRequest, { params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;          // 取出網址中的 id
  journalDb.delete(Number(id));         // 字串轉數字後刪除
  return NextResponse.json({ success: true });
}

5-2　常見 HTTP 狀態碼（回應的「成績單」）

狀態碼意思
200 OK成功
201 Created成功並新增了資料
400 Bad Request前端送來的資料有問題（例如缺必填）
401 Unauthorized沒登入 / 沒權限
404 Not Found找不到資源
500 Internal Server Error後端自己出錯了

Part 6　前端頁面：顯示與送出資料

6-1　顯示資料（Server Component 直接讀資料庫）

Next.js 的頁面預設是「伺服器元件」，可以直接呼叫資料庫函式，把結果畫成 HTML 回傳。簡單又快。

// 檔案：app/page.tsx（首頁）
import { journalDb } from "@/lib/db";
export const dynamic = "force-dynamic"; // 每次都讀最新資料

export default function Home() {
  const journals = journalDb.getAll();   // 直接讀資料庫
  return (
    <div>
      <h1>研究日誌牆</h1>
      {journals.map((j) => (
        <article key={j.id}>
          <h2>{j.date}</h2>
          <p>{j.content}</p>
        </article>
      ))}
    </div>
  );
}

6-2　送出資料（Client Component + 表單）

需要互動（打字、按按鈕）的元件要標記 "use client"。用 useState 記住輸入，用 fetch 把資料 POST 給 API。

"use client";
import { useState } from "react";

export default function NewForm() {
  const [date, setDate] = useState("");
  const [content, setContent] = useState("");

  async function submit(e) {
    e.preventDefault();                     // 阻止表單預設的重新整理
    const res = await fetch("/api/journals", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ date, content }), // 把輸入打包成 JSON
    });
    if (res.ok) alert("新增成功！");
    else alert("失敗，請檢查欄位");
  }

  return (
    <form onSubmit={submit}>
      <input type="date" value={date} onChange={(e) => setDate(e.target.value)} />
      <textarea value={content} onChange={(e) => setContent(e.target.value)} />
      <button type="submit">新增</button>
    </form>
  );
}
💡 把 Part 5 與 Part 6 連起來看，就是 1-4 那條「資料的旅程」：表單(前端) → fetch → API(後端) → journalDb(資料庫) → 回傳 → 更新畫面。理解這一圈，你就懂後端開發的核心了。

Part 7　進階功能怎麼長出來（概念導覽）

有了「前端 → API → 資料庫」的骨架，進階功能都是在這之上加東西。以下說明思路，細節可請 AI 實作。

7-1　登入（Google OAuth）

  • 用 NextAuth 套件接 Google 登入：使用者點登入 → 轉到 Google → 授權後帶回身分。
  • 用 middleware（中介層）擋住 /admin 與寫入型 API：沒登入就導去登入。
  • 用 ALLOWED_EMAILS 清單控管：只有名單內的 email 能進後台。
  • 概念：OAuth 就是「讓 Google 幫你驗證身分」，你不用自己存密碼。

7-2　搜尋與篩選（網址參數 + SQL WHERE）

把搜尋條件放在網址問號後面（例如 ?q=電池&from=2026-06-01），後端讀出來，組成 SQL 的 WHERE 條件。

// 關鍵字 + 日期區間，動態組 WHERE（仍用參數化，安全）
let sql = "SELECT * FROM journals WHERE 1=1";
const params = {};
if (q)    { sql += " AND content LIKE @q"; params.q = "%" + q + "%"; }
if (from) { sql += " AND date >= @from";   params.from = from; }
if (to)   { sql += " AND date <= @to";     params.to = to; }
db.prepare(sql).all(params);

7-3　照片上傳（多檔、拖拉）

  • 前端：用拖拉區 + <input type=file multiple> 收集檔案，存進狀態。
  • 送出：用 FormData 把多個檔案 POST 給上傳 API。
  • 後端：把檔案寫到磁碟，並在 photos 資料表記下檔名與所屬日誌 id。
  • 顯示：用一支路由 /api/uploads/[檔名] 把圖片讀出來給瀏覽器。

7-4　匯出 Word / PDF / HTML

  • 先用同一份資料產生一段「HTML 模板」。
  • HTML：直接輸出檔案；PDF：用 puppeteer（無頭瀏覽器）把 HTML 印成 PDF；Word：用 docx 套件組出表格。
  • 好處：一份內容、三種格式，外觀一致。

8-2　為什麼要改造才能上雲（重要觀念）

很多雲端平台（serverless）的硬碟是「唯讀且暫時」的，重新部署就清空。所以本機能跑的 SQLite 檔、上傳的照片，在那種平台上會消失。解法有兩種：換成雲端資料庫＋雲端儲存，或選一個「有持久磁碟」的平台。我們選後者：Railway。

8-3　Railway 部署步驟

  1. Railway → New Project → Deploy from GitHub repo → 選你的 repo（會自動依 Dockerfile 建置）。
  2. 幫服務掛一顆 Volume（持久磁碟），掛載路徑設為 /data —— 資料庫與照片都存這，重部署不會掉。
  3. 在 Variables 設定環境變數：DATA_DIR=/data、AUTH_URL（你的網址）、AUTH_SECRET、Google 憑證、ALLOWED_EMAILS。
  4. Settings → Networking → Generate Domain 產生公開網址。
  5. 到 Google Cloud 建 OAuth 憑證，重新導向 URI 設為「網址 + /api/auth/callback/google」，並加自己為測試使用者。

用 Docker 容器化時，會寫一份 Dockerfile 指定環境（Node、Chromium、中文字型等）。這份設定也很適合請 AI 產生。

8-4　環境變數（secrets）是什麼

環境變數是「放在程式外面的設定值」，例如密鑰、資料庫位置。好處：換環境（本機 vs 正式）只要改設定、不用改程式；機密不會寫死在程式碼裡被看到。

Part 9　vibe coding 心法與除錯

9-1　怎麼跟 AI 來回（迭代）

  • 一次只講一個目標，做完驗證再做下一個，不要一次塞十個需求。
  • 給範例與限制（要用什麼技術、欄位有哪些、不要做什麼）。
  • 出錯時，把「完整錯誤訊息」與「你做了什麼步驟」一起貼給 AI，不要只說「壞了」。
  • 重要決定（公開/私人、要不要付費、刪不刪資料）自己拍板，別全丟給 AI。

9-2　驗證！驗證！驗證！

AI 會犯錯，所以「驗證」是 vibe coding 最重要的習慣。每加一個功能就實際操作一次：新增一筆資料有沒有出現？登入會不會成功？匯出檔打開對不對？不要看到「成功」字樣就相信。

9-3　我們實際踩過的雷（你大概也會遇到）

症狀原因解法
網站開出來 502 Bad Gateway平台給的連接埠(8080)和網域指向的(3000)不一致讓兩邊對齊（改網域埠或設 PORT）
登入出現「已封鎖/invalid_request」AUTH_URL 填錯，混到 localhost改成正確的正式網址再重新部署
Google 不讓登入帳號不在 OAuth「測試使用者」名單到同意畫面把自己的 email 加進測試使用者
終端機中文變亂碼只是終端機顯示編碼問題資料庫存的其實是對的，打開網頁就正常
💡 踩雷是正常的，每個工程師都一樣。差別在於：會看錯誤訊息、會縮小範圍（是前端、後端、還是資料庫那段？）、會一次只改一個地方再測。

結語

你已經走過了一個真實網站的完整生命週期：觀念 → 建立專案 → 設計資料庫 → 寫 API → 做前端 → 加功能 → 部署上線。最重要的不是記住每一行語法，而是理解「資料如何在前端、後端、資料庫之間流動」，以及「如何清楚地指揮 AI、並驗證結果」。接下來，挑一個你想要的新功能，試著自己描述給 AI，一步步做出來吧！
